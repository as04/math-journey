Track C focus: the SWE-interview goldmine

Recursion is one of those topics that splits programmers. Some find it natural; others find it disorienting. The cure is both — practice with simple examples until you trust the recursive call, and visualize the call tree to see why it works.

Today: recursion fundamentals, the infamous Fibonacci, the discovery that naïve recursion is wildly wasteful, and the fix: memoization — your first dynamic programming pattern. Every "DP" interview question is built on this base.

Recursion in one paragraph

A recursive function calls itself on smaller versions of the same problem. Two parts:

  1. Base case(s): the smallest inputs, where you return directly.
  2. Recursive case: reduce the problem to a smaller one (or several) and combine the results.
def fib(n):
    if n <= 1: return n          # base case
    return fib(n - 1) + fib(n - 2)  # recursive case

Try fib(5). It calls fib(4) and fib(3). fib(4) calls fib(3) and fib(2). Soon you're recomputing fib(2) many times. That's the problem.

Watch the exponential explosion

Pick n. Compute fib(n) naively vs. with memoization. Count the function calls. The naïve version is O(φⁿ) ≈ O(1.618ⁿ). The memoized version is O(n).

Try n = 30 with naïve. It takes ~2 million calls and a noticeable amount of time. Memoized is instant. Try n = 35 naïve at your own risk — it can hang your tab.

The memoization pattern

memo = {}
def fib(n):
    if n in memo: return memo[n]
    if n <= 1: return n
    memo[n] = fib(n-1) + fib(n-2)
    return memo[n]

The change is tiny: cache results, look them up before recomputing. Complexity drops from exponential to linear. This is your first dynamic programming algorithm — top-down with memoization.

The bottom-up variant builds the table iteratively from the base cases — no recursion at all:

def fib(n):
    if n <= 1: return n
    a, b = 0, 1
    for _ in range(n - 1):
        a, b = b, a + b
    return b

Same time complexity, but constant space. The bottom-up form is the "production" version.

Why DP works (the two conditions)

  1. Overlapping subproblems. The same subproblem appears many times. (fib(3) is computed many times in naïve fib(10).)
  2. Optimal substructure. The solution to the big problem is built from solutions to smaller subproblems.

Most DP interview questions ask: "Can you spot these two conditions in the problem?" If yes, define state (what does each subproblem represent?) and transition (how do smaller solutions combine into a bigger one?). Memoize.
