Why "counting" is the start of probability

Most "I don't get probability" really means "I can't count the outcomes correctly." Probability for discrete events is just (favorable outcomes) / (total outcomes) — but counting either correctly is the actual hard skill. Today: nail the counting.

Used everywhere: A/B test design, system reliability calculations, password strength, hashing collisions, ML class-imbalance reasoning, and 80% of brain-teaser interview questions.

The three rules of counting

1. Multiplication rule. If choice A has m options and choice B has n options (independent), there are m × n total outcomes. PIN with 4 digits: 10⁴ = 10,000 possibilities.

2. Permutations. Arranging k distinct items in order out of n: P(n, k) = n! / (n−k)!. Order matters.

3. Combinations. Choosing k items out of n, order doesn't matter: C(n, k) = n! / (k! · (n−k)!) = "n choose k". Order doesn't matter.

Permutation vs combination — the test: would (Alice, Bob) and (Bob, Alice) count as different outcomes? If yes → permutation. If no → combination.

Sample spaces and events

A sample space S is the set of all possible outcomes. An event A is a subset of S.

For a single fair die: S = {1,2,3,4,5,6}. Event "rolled an even number": A = {2,4,6}. Probability: |A| / |S| = 3/6 = 1/2.

For two dice: S has 36 outcomes (each die independent). Event "sum is 7" has 6 outcomes — (1,6), (2,5), (3,4), (4,3), (5,2), (6,1). Probability 6/36 = 1/6.
