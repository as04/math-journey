Part 1: Bayes in real life — three scenarios

Scenario A — A/B test significance

You ship an experiment. The "control" converts at 5.0%, the "treatment" at 5.4%. p-value = 0.04. Should you ship?

The frequentist answer: "yes, p < 0.05, statistically significant." The Bayesian-thinking answer is harder: p(H₀ | data) ≠ p(data | H₀). The p-value is the second; what you actually want is the first. If your prior belief in the effect being real was already low (most experiments don't work), even a "significant" p might leave the posterior probability of a real effect at < 50%.

This is why ML teams at Google increasingly use Bayesian A/B testing (often via tools like Croak or Vizier). The math is the disease-test math from Day 10.

Scenario B — Code review heuristic

You see a PR with 500 lines of changes. What's P(this PR has a bug | this PR is large)?

Without numbers: large PRs are more likely to have bugs, so P(bug | large) > P(bug). But by how much? You'd need: base rate of bugs per PR (P(bug)), and P(large | bug) vs P(large | no bug). The shape of the reasoning is Bayesian. The conclusion ("split this PR before merging") is the engineering action that flows from the math.

Scenario C — Spam filter intuition

Naïve Bayes spam filter: P(spam | words) ∝ P(words | spam) · P(spam). "Naïve" because we assume words are independent given the class (which is wrong, but works well in practice). The multiplication of many small likelihoods can underflow floating-point — so real implementations sum log-probabilities instead. (Another reason log-thinking matters in CS.)

Part 2: Project Euler #3 — Largest prime factor

"The prime factors of 13195 are 5, 7, 13 and 29. What is the largest prime factor of the number 600851475143?"

Step 1 — Naïve approach (and why it dies)

def largest_prime_factor_naive(n):
    largest = 1
    for i in range(2, n + 1):
        if n % i == 0:
            # check if i is prime by trial division
            is_prime = all(i % j != 0 for j in range(2, int(i**0.5)+1))
            if is_prime:
                largest = i
    return largest

For N = 600,851,475,143, this checks ~6×10¹¹ values. Each check itself is non-trivial. Hours, maybe days.

Step 2 — The math insight

To find the largest prime factor, divide out small primes first:

  1. While n is divisible by 2, divide. Then n has no more factor of 2.
  2. Try odd factors 3, 5, 7, ... up to √n. Each time you find one, divide it out as many times as possible. (This guarantees the factor is prime — anything that divides n at this point and is ≤ √n must be prime, since composites would have already been split.)
  3. If after all this n > 1, that remaining n is itself prime — and it's the largest factor.

Why stop at √n? Because if n has any factor > √n, it has a corresponding factor ≤ √n. So if you've removed all factors ≤ √n, what's left is either 1 or a single large prime.

Step 3 — Optimized solution

def largest_prime_factor(n):
    largest = 1
    # factor out 2s
    while n % 2 == 0:
        largest = 2
        n //= 2
    # factor out odd primes up to sqrt(n)
    f = 3
    while f * f <= n:
        while n % f == 0:
            largest = f
            n //= f
        f += 2
    # whatever remains (if > 1) is itself prime
    if n > 1:
        largest = n
    return largest

print(largest_prime_factor(600851475143))  # 6857

Complexity: O(√n) in the worst case. Runs in milliseconds for the PE problem. From "days" to "milliseconds" via one math insight.
